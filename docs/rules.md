# ゲームルール

## デッキ構築ルール

### 枚数制限

| 項目 | 制限 |
|---|---|
| デッキ枚数 | ちょうど 30 枚（`DeckModel.MaxCards`）でのみ対戦可能 |
| 枚数オーバー | 30 枚を超えると「30枚を超えています」が赤字表示され、枚数カウンターも赤くなる |
| Home での対戦ブロック | 枚数が 30 枚でないときにトーストを表示して遷移をブロック |
| 同名カードの枚数制限 | 同一カード（ID 基準）は 3 枚まで（`DeckRuleModel.MaxCopiesPerId`）。4 枚目を追加しようとすると「同名カードは3枚までです」トーストを表示して拒否（ドラッグ／右クリック／＋ボタンすべて）。保存済みデッキの読み込みは制限対象外 |
| 制限の ON/OFF | Editor 再生時のみ Home シーン左上のトグル（`DeckRuleModel.LimitSameCards`）で切り替え可能。トグルの状態は `EditorPrefs` に保存され、再生をまたいで前回の状態を引き継ぐ。ビルドにはトグルが存在せず `EditorPrefs` も読まないため常に ON。CPU デッキ構築は Home トグルに依存せず常に ON |

### 構築手順

1. Home シーンから「デッキ構築」を押すとデッキ構築シーンへ遷移する
2. 左パネルのカード一覧から選択してデッキに追加する（フィルタボタン横に「枚数 N/30」が常時表示。コスト表示はなし。同一カードは 3 枚まで）
3. 30 枚を超えると「30枚を超えています」が赤字で表示される
4. 「もどる」を押すと Home シーンに戻る（変更は自動保存済み）

### その他

- デッキは DeckModel に保持され、DeckBuilder → Main 遷移をまたいで引き継がれる
- CPU デッキは `CpuDeckSO`（ScriptableObject・Addressables キー `"Card/CpuDeck"`）から読み込む
  - カードIDリストが 1 件以上の場合：`CardDatabase.BuildDeck` を呼び出したカード群をシャッフルして使用する
  - カードIDリストが空の場合：カードDB全カードをシャッフルした全枚数をフォールバックとして使用する
- CPU デッキの編集は Unity Editor 専用の `CpuDeckBuilder` シーンで行う（タイトル画面の「CPU デッキ編集」ボタン、`#if UNITY_EDITOR` でのみ表示）。編集内容は `CpuDeckSO`（`Assets/Data/CpuDeck.asset`）に保存され、ビルドに含まれる

## ゲーム開始時フロー（1回のみ）

```
マリガン選択（プレイヤー）
  ↓ 手札内容に関わらず YES/NO モーダルを表示（確認中も手札カードをクリックしてカード詳細を閲覧可能）
  ↓ YES → 「マリガン」告知 → 手札を全てデッキへ返却 → リシャッフル → 再ドロー
  ↓ NO → 手札をそのまま継続
  ↓ （1回限り）
マリガンチェック（CPU）
  ↓ 初期手札にキャラカードがなければ手札を全てデッキへ返却 → リシャッフル → 再ドロー（告知なし）
  ↓ （オンライン時は相手プレイヤーの決定を受信してアニメーション表示。下記参照）
  ↓
先攻後攻コイントス
  ↓ ランダム（オンラインはホスト決定）で先攻・後攻を決定
  ↓ コインフリップ演出 → 「先攻！」または「後攻！」を表示
  ↓ （補正ドローなし。先攻有利はターン1のドローなしで補正する）
  ↓
通常ターンループへ（先攻プレイヤーのドローフェーズから開始）
```

**オンライン時のマリガン**: 両プレイヤーが独立して YES/NO モーダルで選択する。自分の選択後に `NGS_Mulligan` メッセージで決定を相手に送信し、相手の決定を受信するまで「対戦相手を待っています」を表示する。相手が YES を選択していた場合はアニメーション演出を表示する。マリガン要否の判定はホストが行わず、各プレイヤーが自律的に決定する。

## ターンフロー

ターンは先攻プレイヤーから交互に繰り返す。

```
【アクティブプレイヤーのターン開始】
ターン開始時効果（ドロー前）
  ↓ 場のキャラの OnTurnStart 効果を並び順に発動
  ↓ プレイ済みの永続イベント（EventCardTrigger.OnTurnStart）を墓地からせり出して発動
  ↓
ドローフェーズ
  ↓ アクティブプレイヤーのみデッキ上から3枚ドロー（デッキが 0 枚の場合はスキップ）
  ↓ ただし先攻プレイヤーの初手（通算ターン1）はドローなし（先攻有利の補正）
  ↓ ドロー枚数は効果で増減する: EventType.DrawSkipNext は予約された側のドローを 0 枚に、EventType.DrawNextTurnStart は予約枚数を上乗せ（いずれも消費。詳細は event.md）
  ↓ オンライン時: アクティブプレイヤーが NGS_Draw を送信、相手がアニメーション再生（0枚でも同期のため送信）
  ↓
メインフェーズ
  ↓ アクティブプレイヤーは EndButton を押すまで、枚数・攻撃回数の制限なく以下を繰り返せる
  ↓   キャラを出す: 手札のキャラカードをフィールドに表向きで配置 → 手札からコスト枚数選択して墓地へ
  ↓   イベントを使う: 手札のイベントカードをフィールドへ → 即時解決（手札コスト選択 → 効果 → 墓地）
  ↓   攻撃する: フィールドのキャラから矢印をドラッグして相手キャラへ攻撃（各キャラ1回まで）
  ↓ EndButton（画面右）でターン終了
  ↓
ターン終了 → 相手のターンへ
```

- 例外: `EventType.ExtraTurn` の効果が発動したターンは、相手にターンを渡さず**同じプレイヤーがもう一度ターンを行う**（ターン開始時効果→ドロー→メインフェーズ）。詳細は [event.md](event.md) を参照。

## メインフェーズの詳細

アクティブプレイヤーは **EndButton を押すまで、枚数・攻撃回数の制限なく**以下のアクションを繰り返せる。各アクション後も同じメインフェーズが続く。

### アクション選択肢

| アクション | 操作方法 | 処理 |
|---|---|---|
| キャラを出す | 手札のキャラカードをフィールドへドロップ → コスト選択 → OK ボタン | フィールドに表向きで配置 → 手札からコスト枚数を選択して墓地へ送る（フィールドのキャラは9体まで。満杯時はドロップ拒否） |
| イベントを使う | 手札のイベントカードをフィールドへドロップ → コスト選択 → OK ボタン | フィールドへ配置 → 即時解決（手札コスト選択 → 効果 → 墓地）|
| 攻撃する | フィールドのキャラから矢印をドラッグして相手キャラへ | 攻撃実行（下記「攻撃の詳細」参照）|
| ターン終了 | EndButton（画面右） | 自ターンを終了して相手のターンへ |

- コスト 0 のカードはドロップ後すぐ OK ボタンが有効になる
- Back ボタンでコスト選択をキャンセルしてカードを手札に戻せる
- AutoOk が有効な場合、コスト 0 はドロップ直後に自動で OK、コストありは必要枚数を選択した時点で自動で OK

### 攻撃回数と召喚酔い

- **各キャラの攻撃はターンに1回まで**（`_attackedThisTurn` で管理。攻撃済みキャラは矢印を出せない）
- **召喚酔い**: このターンに登場したキャラ（配置・召喚・Switch / Evolve での配置）はそのターン攻撃できない。次の自ターンから攻撃可能。判定は自メインフェーズ開始時のフィールド面子のスナップショット（`_playerSeasonedChars` / `_opponentSeasonedChars`）で行う
- **速攻**: `CharacterCardData.Haste`（SO のチェックボックス）を持つキャラは召喚酔いしない＝場に出したターンから攻撃できる（`CanCharAttack` で seasoned 判定をバイパス。1ターン1回の攻撃制限は維持）。速攻持ちはカードと詳細モーダルに SpeedIcon を表示
- このターン攻撃できる自キャラはシアン色でハイライト表示される（USS `attackable-char`）
- 攻撃できる自キャラが1体以上いる場合、攻撃可能な相手キャラを赤〜オレンジ系でハイライト表示する（USS `attack-target-char`、攻撃側のシアンと区別）。「いずれかの攻撃可能な自キャラが攻撃できる対象」を守護・飛行を考慮して強調する（例: 守護がいる場合は守護持ちのみだが、飛行を持つ自キャラがいればそれが狙える対象も加わる）。メインフェーズのループで毎回再計算される
- オンラインでは EndButton 押下を Pass アクションとして相手へ送り、ターン終了の合図に使う（追加メッセージは不要）

### CPU のアクション選択順

毎ループ以下を判定し、いずれも該当しなくなるまで繰り返す（該当なしでターン終了）。攻撃の候補は「このターン未攻撃かつ召喚酔いしていないキャラ」のみ。

1. **lethal デッキ攻撃**: デッキを直接攻撃でき（`CanAttackDeck`）、ATK がプレイヤーのデッキ枚数以上の攻撃者がいれば、デッキ攻撃で引き切らせて勝利を狙う（最高ATKを選ぶ）
2. **キャラ攻撃**: 攻撃可能なキャラがいて、かつ相手フィールドに合法な攻撃対象がいれば攻撃（`CanAttackChar` で守護・飛行を考慮し、合法な対象を持つ攻撃者の中で最高ATK→対象は最低ATKを選ぶ）
3. **チップミル（デッキ攻撃）**: 合法なキャラ攻撃対象がいない場合、デッキを直接攻撃できる攻撃者がいればデッキを削る（最高ATKを選ぶ）
4. 手札に**コストを支払えるキャラカード**があれば配置
5. 手札に**コストを支払えるイベントカード**があれば使用
6. 上記いずれもなければターン終了

> 配置・使用の対象は「自身を除いた手札の支払い可能量（`CostPaymentValue` 合計）がコスト以上」のカードに限る（`MainPresenter.CpuCanAffordCost`）。ローカルプレイヤーと同じくコストの踏み倒しはできない。`NextCardCostFree` 武装時とコスト 0 のカードは常に支払い可能。

### 攻撃の詳細

- **一方向攻撃のみ**（カウンター攻撃なし）
- 攻撃対象: 相手フィールドの表向きキャラ、または**相手デッキ**（矢印で選択）
- キャラ攻撃: 相手にキャラがいない場合はキャラへの攻撃は不可
- **デッキへの直接攻撃（ミル）**: フィールドのキャラから矢印を**相手デッキ**へドラッグすると、攻撃キャラの **ATK 枚**だけ相手デッキの上から墓地へ送る（デッキ枚数が ATK 未満なら全枚数）。ATK 0 はシールドアイコン表示で不発。各キャラ1回までの攻撃制限・召喚酔いはキャラ攻撃と共通（デッキ攻撃もそのキャラのターンの攻撃を消費する）。デッキ攻撃で相手デッキを0枚にすると相手はデッキ切れで敗北（「勝敗条件」参照）。相手デッキが0枚のときは攻撃できない。攻撃できる自キャラがいる間は相手デッキを赤〜オレンジ系でハイライト（USS `deck-view--attack-target`）
- **守護（タウント）**: 相手フィールドに**守護**を持つキャラがいる間は、守護持ちキャラにしか攻撃できない（守護以外のキャラ・**相手デッキ**への攻撃は不可）。守護が複数いればそのいずれか。守護が破壊されれば通常の対象選択に戻る。守護持ちキャラはカードの HP アイコンの下に ShieldIcon を表示。守護は `CharacterCardData.Guardian`（SO のチェックボックス）で設定（攻撃のみを制限し、`DamageEnemy` 等のカード効果の対象選択には影響しない）
- **飛行**: `CharacterCardData.Flying`（SO のチェックボックス）を持つキャラは、**守護を無視**して攻撃対象（相手キャラ・相手デッキ）を選べ、かつ**飛行を持つキャラからしか攻撃されない**（飛行なしキャラは飛行キャラを攻撃不可）。飛行同士は攻撃可能。守護かつ飛行を持つキャラは、飛行でしか攻撃できず、かつ守護として攻撃を強制する（飛行なし攻撃者は対象がなくなり実質攻撃不可）。攻撃のみに作用し、`DamageEnemy` 等のカード効果の対象選択には影響しない。判定はキャラ攻撃が `CanAttackChar`・デッキ攻撃が `CanAttackDeck` に中央化し、プレイヤー攻撃・CPU・攻撃ハイライトで共用する。飛行持ちはカードと詳細モーダルに FlyIcon を表示

### ダメージ計算

- ダメージ = `ATK`（防御力なし）
- ダメージ > 0: 攻撃対象キャラの現在 HP からダメージ値を減算する（累積ダメージ）
- ダメージ = 0: シールドアイコンをフローティング表示して攻撃を終了

### キャラ破壊

- 攻撃を受けたキャラの現在 HP からダメージ値を減算する（累積ダメージ）
- HP が **0 以下**になったキャラカードを墓地へ送る
- HP の残量は `CardView.CurrentHp` で管理し、カード上の HP ラベルにリアルタイムで反映される
- キャラをフィールドに配置するたびに HP は最大値（`CardData.Hp`）にリセットされる（Switch で手札に戻して再出しした場合も同様）

カードの効果種別（`EventType`）の一覧と、キャラ・イベントカードへの設定方法は [event.md](event.md) を参照。

## コストシステム

全カードは `Cost` 値を持つ。カードをプレイするとき、コスト分だけ**手札から選んで墓地**へ送る。通常カードは1枚＝1コストだが、`CostBoost` カードは1枚で設定値（例: 2）コスト分として数える（コスト判定は枚数ではなく各カードの `CostPaymentValue` の合計で行う）。ただし `CostBoost` の倍化は**属性連動**：CostBoost カードの属性がプレイするカードの属性と一致するときのみ設定値分で、それ以外の属性のコストには通常どおり1として数える（白も一般属性として扱い、白 CostBoost は白のコストのみ倍化）。

| カード種別 | コストを払うタイミング |
|---|---|
| キャラカード（メインフェーズ） | ドロップ直後にコスト選択UI表示 → コスト選択後の OK ボタンで配置確定と同時 |
| イベントカード（メインフェーズ） | ドロップ直後にコスト選択UI表示 → コスト選択後の OK ボタンで確定と同時（即時解決） |

- コスト 0 のカードはコスト選択をスキップする（ドロップ後すぐ OK が有効）
- プレイするカード自身を除いた手札の支払い可能量（各カードの `CostPaymentValue` 合計）がコストに満たない場合はドロップを拒否し「コストが払えません」トーストを表示
- コスト選択 UI: カードをドロップするとコスト選択モードになり、クリックして選択（選択済カードは透明度低下）し OK で確定（1 回押すだけ）。選択カードの `CostPaymentValue` 合計がコストに達すると OK が有効
- CPU はプレイ前に支払い可能性を確認し（`CpuCanAffordCost`）、自身を除いた手札の `CostPaymentValue` 合計がコストに満たないカードは出さない。支払うカードは属性制約を満たすよう自動選択（同属性を優先して1枚確保、残りはコスト値の合計がコストに達するまで）
- オンライン対戦では払ったカードの ID をネットワーク送受信して双方の相手墓地表示を同期する

### 属性制約

全カードは `CardAttribute`（Red / Blue / Green / Yellow / Black / Purple / White）を持つ。白を含むすべての属性を同列に扱い、コスト支払い時に以下の制約がある：

| プレイするカードの属性 | コスト選択の制約 |
|---|---|
| 全属性（White を含む） | 選択したコストカードの中に「**同じ属性**」が **1 枚以上**必要 |

- 制約を満たさない場合は OK ボタンが無効になり、「「赤」のカードを1枚以上含めてください」のような警告を表示する
- White は万能コストではない。白カードは他属性のコストの属性要件を満たさない（コスト枚数の数合わせとしては選べる）
- 白カードをプレイするときは、コストに白カードを最低1枚含める必要がある

## 勝敗条件

属性に依らない**共通の3条件**で勝敗が決まる（ゲーム開始時から両プレイヤーに適用。`WinRule` / 勝因種別は `WinReason`）。いずれか1つでも満たした瞬間に決着する。

### 1. デッキ切れ（`WinReason.DeckOut`）

あるプレイヤーのデッキが**0枚**になると、その**相手**が勝利する（デッキを失った側が敗北＝デッキアウト）。デッキが0になる経路は2つ：自分のドローで引き切る、または**相手のデッキ攻撃（ミル）で削り切られる**（「攻撃の詳細」参照）。

| 項目 | 内容 |
|---|---|
| 勝敗 | デッキが0枚になったプレイヤーが敗北し、相手が勝利 |
| 判定タイミング | デッキが減る処理の直後に `CheckDeckOutWin` で判定（`WinRule.IsDeckOut`）：各ドロー処理（ドローフェーズ・Drawイベント効果）の直後、およびデッキ攻撃のミル直後 |

- オンライン対戦では既存のドロー同期（`NGS_Draw`）とデッキ攻撃同期（`NGS_MainAction` の `targetsDeck`）を流用し、デッキ枚数・デッキ順が両クライアントで同期されるため追加のネットワークメッセージなしで同じ勝敗に到達する

### 2. 勝利点20（`WinReason.VictoryPoints`）

勝利点が **20（`WinRule.VictoryPointsToWin`）以上**になった側が勝利する。勝利点カウンター（MedalIcon + 数字）は**ゲーム開始時から両者に常時表示**される（`VictoryPointsView`）。

| 項目 | 内容 |
|---|---|
| 表示位置 | 自分 → **自分フィールドの左下** ／ 相手 → **相手フィールドの右上**。開始時から 0 で表示 |
| 加点 | 全カード共通の `VictoryPointBonus` 付帯値（効果＝EventType とは独立した加算フィールド）で、効果解決時に発動した側の勝利点に加算する。加点カードは緑属性で作る（緑以外では付帯値は常に 0）。「勝利点を得るだけ」は `EventType=None` ＋ 付帯値で表現し、「効果＋勝利点」も付帯値の併用で作れる。`EventType.GainVPPerGreenGrave` は固定値ではなく、発動した側の墓地にある緑属性カードの枚数だけ加算する（墓地に緑がなければ加点 0） |
| 勝敗 | 勝利点が 20 以上になった瞬間にその側が勝利（`AddVictoryPoints` 内で判定） |
| 演出 | 発動カードの上に MedalIcon フロート → 勝利点カウンターで「+N」とメダルの弾み → 数字カウントアップ。演出後に勝利判定 |

- 勝利条件は共通だが、勝利点を**得る手段**は緑属性カードのみ（`VictoryPointBonus` は緑以外では 0）。属性ゲートはなく、緑を使わない側にも最初からカウンターが表示される
- オンライン対戦では既存のプレイ同期（`NGS_MainAction`）で効果が両クライアントに伝わり、勝利点も決定的に加算されるため追加のネットワークメッセージは不要

### 3. キャラ8体（`WinReason.FieldChars`）

自フィールドにキャラを **8体（`WinRule.FieldCharsToWin`）同時**に並べた側が勝利する。

| 項目 | 内容 |
|---|---|
| 勝敗 | 自フィールドのキャラ数が 8 体に達した瞬間にその側が勝利（破壊されて減ると再び 8 体並べ直す必要がある） |
| 判定タイミング | キャラ配置・召喚（通常配置・SummonChar・SummonFromDeckByKeyword・Switch / Evolve での配置）の直後に `OnCardPlayed` → `CheckFieldCharsWin` で判定 |
| フィールド上限 | フィールドのキャラ上限は 9 体（`FieldView.MaxCharacters`）のため 8 体は到達可能 |

- オンライン対戦では配置・召喚が `NGS_MainAction` と決定的な召喚処理で両クライアントに同期されるため、キャラ数判定も対称に成立する（追加のネットワークメッセージ不要）

### 勝敗演出

勝敗決定時に画面中央にオーバーレイ演出を表示し、ホームに戻るボタンが出現する。

| 決着の種類 | 表示テキスト | 紋章 | テキスト色 |
|---|---|---|---|
| デッキ切れ（`WinReason.DeckOut`） | 「デッキ切れ勝利」「デッキ切れ敗北」 | GraveIcon | 青系（敗北側は暗め） |
| 勝利点（`WinReason.VictoryPoints`） | 「勝利点勝利」「勝利点敗北」 | MedalIcon | 金系（敗北側は暗め） |
| キャラ8体（`WinReason.FieldChars`） | 「制圧勝利」「制圧敗北」 | AttackIcon | 橙系（敗北側は暗め） |
| 降参・タイムアウト・引き分け | 「YOU WIN」「YOU LOSE」「DRAW」 | なし | 勝=金 / 敗=赤 / 分=灰 |

- 降参勝ちは「対戦相手が降参しました」、自分の降参は「降参しました」をサブラベルに表示する
- 勝利時は花火演出、敗北時は次回ホームで雨演出（`ShouldRainOnNextHome`）
- オーバーレイの暗幕は `PickingMode.Ignore` でクリックを透過するため、勝敗演出後も**フィールドの表向きキャラカードをクリックしてカード詳細を閲覧できる**（再戦・ホームのボタンは通常どおりクリック可能）

#### 再戦

勝敗演出のオーバーレイには「再戦する」「ホームに戻る」の2ボタンが並ぶ（勝ち・負け・降参いずれの決着でも表示）。

- **オンライン**: 「再戦する」を押すと `NGS_Rematch` を送信し「対戦相手を待っています...」を表示。双方が押す（自分の希望と相手の希望が揃う）と両者が Main シーンを再ロードして新規対戦を開始する。相手が先に希望すると「対戦相手が再戦を希望しています」を表示。待機中に相手が退出すると「対戦相手が退出しました」を表示して再戦ボタンを消す
- **CPU 戦**: 「再戦する」を押すと相手待ちなしで即再戦
- 再戦はデッキ（`DeckModel`）を維持したまま再配牌し、先攻後攻はホストが再抽選する。NGO セッションは維持されるため再ハンドシェイク（`PrepareDecksAsync`）で開始する。詳細は [networking.md](networking.md#再戦ngs_rematch) を参照
