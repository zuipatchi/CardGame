コードを読んでおかしなところ（バグ・規約違反・設計の逸脱・パフォーマンス上の問題）を検出して報告する。読み取り専用で、修正はしない。

手順:
1. Explore エージェントを観点別に並列で spawn して調査させる（1観点1エージェント）
   **調査範囲は `Assets/Scripts/` 配下全体**（変更差分だけでなく既存コードも横断的に見る）。ユーザーが特定ファイル・ディレクトリを指定した場合のみそこに絞る
   各エージェントに対象範囲と CLAUDE.md の規約を伝える
   重複検出（duplicate-check）・ファイル分割（split-check）はスコープ外。被る指摘はしない

   **観点A: バグ・不具合**
   - null 参照（`[Inject]` 漏れ・`Q<T>` の要素名タイポ・null チェックなしの参照）
   - 非同期の例外握りつぶし（`async void`・await し忘れ・`catch {}`・`OperationCanceledException` 漏れ）
   - R3 の購読リーク（`Subscribe` の戻り値を `AddTo` / `Dispose` していない）
   - `CancellationToken` の伝播漏れ、画面破棄後に走り続ける処理
   - 競合・順序依存（初期化前アクセス・`await` を跨いだ前提崩れ・二重実行ガード欠如）
   - 境界・分岐の取りこぼし（空コレクション・0枚・最大枚数・switch の default 漏れ）

   **観点B: 規約違反（CLAUDE.md）**
   - 命名（フィールド `_camelCase`、型/メソッド/プロパティ/定数 `PascalCase`）違反
   - `var` の使用、アクセス修飾子の欠如、波括弧の省略・開き波括弧の同一行配置
   - DI 禁止事項（`GameObject.Find` / `FindObjectOfType` / static 経由の依存取得）
   - `Resources.Load` の使用（Addressables を使うべき）
   - uGUI の使用、スタイルのインライン指定（USS に置くべき）
   - USS 非サポートプロパティ（`gap` / `row-gap` / `column-gap`・`-unity-background-scale-mode: scale-to-fill`）
   - using が名前空間の内側に書かれている

   **観点C: 設計・アーキテクチャの逸脱**
   - 単方向フロー違反（Presenter が Model を経由せず状態を直接書き換え・View が Model を直接参照）
   - 責務の混在（Model にUI処理・Presenter にロジックの塊・Service が通信/変換/状態を全部持つ）
   - DI の誤用（LifetimeScope への登録漏れ／二重登録・スコープ違い）
   - ReactiveProperty の使いどころ違い

   **観点D: パフォーマンス**
   - `Update` / 毎フレーム処理内の重い処理（LINQ アロケーション・`Q<T>` 呼び出し・文字列生成）
   - キャッシュすべき参照の都度取得（`root.Q<T>("name")` を毎回呼ぶ → Awake でキャッシュ）
   - 不要なアロケーション（ループ内ラムダキャプチャ・`ToList()` / `ToArray()` 乱用）
   - Addressables / 音声ロードの多重ロード・解放漏れ

   各問題について「ファイルパスと行番号」「観点」「深刻度（高＝動作不良/クラッシュ ／ 中＝潜在バグ/明確な規約違反 ／ 低＝改善提案）」「何が問題か（1〜2文）」「確信度（確実／要確認）」を報告させる

2. 各エージェントの報告を統合し、深刻度順に一覧でユーザーへ報告する（修正はしない）
   - 「要確認」の項目は、なぜ問題かもしれないかを補足する
   - 問題なしの観点があれば明記する
   - 修正したい項目があればユーザーが指示する。「直して」と言われたら初めて修正に着手する（CLAUDE.md の規約を守る）

注意:
- **読み取り専用**。報告で必ず止まる。勝手に Edit / コミットしない
- 重複・ファイル肥大は指摘しない（duplicate-check / split-check の担当）
- 自動テストの不在は問題として報告しない（このプロジェクトはテストを書かない方針）
- 確信が持てない指摘は「要確認」と明示し、断定・捏造しない。指摘は具体的に（ファイル・行・理由）
